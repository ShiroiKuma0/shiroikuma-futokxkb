# FUTO Keyboard — customized fork build skill

The user maintains a customized build of [FUTO Keyboard](https://github.com/futo-org/android-keyboard) on Android, run on a Huawei Mate XT alongside the official FUTO Keyboard. The fork's purpose: a distinct `applicationId` + label so it installs side-by-side, plus a growing set of feature changes the user builds, tests, and may publish as "plus" releases from their GitHub fork.

The fork model is identical to the user's other Android forks (AppManager, Jami, ArcaneChat, FairEmail, SimpleX): fork on GitHub, a mirror branch tracking upstream, a `custom` branch carrying the user's commits, rebased onto each new upstream release tag, force-pushed to the fork. The user builds locally and sideloads; builds are signed with a stable per-fork keystore so reinstalls land in place.

## Why build `stable`, not `unstable`

Upstream declares three product flavors on dimension `buildType`: `unstable` (upstream default, `getIsDefault().set(true)`), `stable`, `playstore`. `unstable` appends `applicationIdSuffix ".unstable"` + `versionNameSuffix "-unstable"` + overrides the label to "FUTO Keyboard [Dev Build]" (`java/unstable/res/values/strings-appname.xml`). `playstore` appends `.playstore`. Only **stable** appends nothing and inherits our base label — so it yields a clean `shiroikuma.futokxkb` / `0.1.28+1` / `白い熊 FUTO kxkb`. Always build `:assembleStableRelease`. The stable flavor's update-checking BuildConfig has `UPDATE_CHECKING_NETWORK = false`, so it will not phone home.

## Side-by-side install is safe (no collisions)

Coexistence is decided purely by `applicationId` (`shiroikuma.futokxkb` vs official `org.futo.inputmethod.latin`). The `namespace` stays `org.futo.inputmethod.latin` — it is build-time only (R/BuildConfig package) and never seen by the OS at install. All three FUTO content providers use `${applicationId}` authorities (`.font`, `.clipboard`, `.androidx-startup`), so they auto-differ; there are no hardcoded authorities and no custom `<permission>` declarations. So FUTO + the fork install as two separate keyboards. Do NOT install the fork over official FUTO (different signing keys → Android refuses); they coexist as distinct packages.

### Commit 2 — `Quieter build logs`

Suppress the wall of build warnings **at the source** (committed), the way the AppManager / Jami forks do it. FUTO emits hundreds of clang `C/C++:` native warnings (sign-compare, cast-align, float-equal in ggml/llama/protobuf/sentencepiece), Kotlin `w:` deprecations, and Gradle/javac notices that bury the lines that matter. Everything the *compilers* emit is killed at source (below); only three *tooling*-emitted blocks with no source toggle get a narrow pipeline filter (see "Tool-emitted warnings" below).

**`build.gradle` `defaultConfig`** — silence native clang warnings (`-w`):

```diff
         vectorDrawables.useSupportLibrary = false
         ndk { abiFilters 'arm64-v8a' }
+        externalNativeBuild {
+            cmake {
+                cppFlags '-w'
+                cFlags '-w'
+            }
+        }
```

(Merges with the existing android-level `externalNativeBuild { cmake { path 'native/jni/CMakeLists.txt' } }`.)

**`build.gradle` `kotlinOptions`** — silence Kotlin `w:` warnings:

```diff
     kotlinOptions {
         jvmTarget = '1.8'
+        suppressWarnings = true
     }
```

(**Property assignment `= true`, not the method-call form `suppressWarnings true`** — the old `kotlinOptions` DSL parses the space form as a method call `suppressWarnings(true)`, which does not exist and fails configuration with "Could not find method suppressWarnings()".)

**`build.gradle`** — silence the javac deprecation notes; add this block at script top level (outside `android {}`, e.g. at the end of the file):

```groovy
tasks.withType(JavaCompile).configureEach {
    options.compilerArgs << '-Xlint:none' << '-nowarn'
    // javac's mandatory deprecation NOTES ("uses or overrides a deprecated API" /
    // "Recompile with -Xlint:deprecation for details") are NOT suppressed by -Xlint:none
    // or -nowarn (verified empirically on JDK 21 — those flags are exactly what *triggers*
    // the summary note). -XDsuppressNotes (internal javac flag) is the only thing that
    // silences them; it suppresses NOTE-kind diagnostics only, errors still print.
    options.compilerArgs << '-XDsuppressNotes'
    options.deprecation = false
}
```

**`gradle.properties`** — append:

```properties
android.javaCompile.suppressSourceTargetDeprecationWarning=true
org.gradle.warning.mode=none
```

Errors are unaffected by any of the above — `-w`/`-nowarn`/`suppressWarnings`/`-XDsuppressNotes` drop warnings and notes only, so build failures (native `error:`, Kotlin `e:`, Gradle task failures, `BUILD FAILED`) still print.

#### Tool-emitted warnings with no source toggle — narrow pipeline filter

The source-level flags above cover everything the **compilers** emit. Three remaining warning blocks come from build *tooling* (NDK/CMake, the manifest merger, AAPT2) and have **no source-level suppression** — they are not compiler diagnostics, so no `build.gradle`/`gradle.properties` flag silences them:

1. **`[CXX5304] This version only understands SDK XML versions up to 3 but an SDK XML file of version 4 was encountered.`** — NDK/CMake bridge vs a newer SDK `package.xml`. Benign; appears twice during native configure.
2. **Manifest-merger no-ops** — `…/java/stable/AndroidManifest.xml Warning:` + `uses-permission#…ACCESS_NETWORK_STATE was tagged at AndroidManifest.xml:0 to remove other declarations but no other declaration present` and `provider#org.acra.attachment.AcraContentProvider@android:authorities … to replace … but no other declaration present`. The stable flavor manifest carries `tools:node="remove"`/`"replace"` guards (to strip ACRA/network-state) that have nothing to act on in this merge → harmless warnings. (Editing the manifest to drop the guards is *not* worth it — they're intentional safety for other merge inputs.)
3. **AAPT2 resource-shrink** — `warn: removing resource shiroikuma.futokxkb:string/… without required default value.` (credits_title, version_text, several typing/size settings strings). Locale-only strings with no default value, dropped during resource processing. Benign.

Because there is no source flag, these — and **only** these — are dropped with a tight, explicit `grep -v` on the gradle output, keeping the success signal via `PIPESTATUS` so errors and `BUILD FAILED` are never hidden. This is the single sanctioned exception to "no output grep filter"; do not widen the pattern. In the build pipeline, replace the bare `r ./gradlew …` invocation with:

```bash
  # benign tool warnings with no source toggle (NDK CXX5304, manifest-merger no-op
  # remove/replace, AAPT2 locale-string removal). Tight pattern — do not broaden.
  NOISE='CXX5304|AndroidManifest\.xml Warning:|was tagged at AndroidManifest\.xml:0 to (remove|replace) other declarations but no other declaration present|^warn: removing resource .* without required default value'
  BRANCH_NAME='shiroikuma' VERSION_NAME="$our_name" VERSION_CODE="$our_code" \
    ./gradlew :assembleStableRelease --console=plain 2>&1 \
    | sed -E "/$NOISE/d"
  [ "${PIPESTATUS[0]}" = 0 ] && build_ok=1
```

`PIPESTATUS[0]` is gradle's own exit status (not `sed`'s), so success/failure detection is unaffected by the filter. Everything except the three listed blocks still prints verbatim.

## Versioning (local counter + env injection — no commit needed)

Upstream's `build.gradle` already derives the version from environment variables, falling back to git: `getVersionCode()` reads `VERSION_CODE`, `getVersionName()` reads `VERSION_NAME`. So **no build.gradle edit is needed for versioning** — we inject the env vars at the `gradlew` call, like the Jami fork injects `-PshiroikumaBuild`.

Scheme (matches AppManager/Jami):
- **Base name** = the FUTO release being tracked, e.g. `0.1.28` (= the rebase target tag, leading `v` stripped).
- **Base code** = FUTO's own code for that release = `git rev-list --first-parent --count <tag>` (count the **tag**, not `master` — FUTO's CI counts master at release time, and master moves on afterward; counting the tag reproduces the released code, e.g. `0.1.28` → `11654`).
- **N** = per-build iteration, from a local counter file `~/tmp/.shiroikuma_futokxkb_build` (one line: `<base_name> <N>`). Increment on each **successful** build; **reset to 1 when the base changes** (new upstream tag). Consumed only on success.
- **versionName** = `<base_name>+<N>` → `0.1.28+1`, `0.1.28+2`, …
- **versionCode** = `<base_code> * 10000 + N` → `116540001`, `116540002`, … (9999 builds of headroom per base, always above the next upstream bump). `BRANCH_NAME='shiroikuma'` is also exported so the three `BRANCH_NAME env variable not set` fallback lines never print.

The build block keeps N in the counter and bumps it itself — never hardcode `buildN`. `+` is legal on ext4/sdcard FAT/`adb push`/GitHub assets; leave it unescaped.

## Signing

The release buildType has a `signingConfig` that reads `keystore.properties` (falling back to the repo's debug keystore if absent), so AGP produces an **already-signed, zipaligned** APK — no separate `apksigner`/`zipalign` step (unlike the Jami/AppManager/ArcaneChat forks). We use a dedicated keystore (created once) so the user owns the key and avoids the repo keystore's MD5withRSA cert. `keystore.properties` is **regenerated by the build block on every run** (heredoc) so it survives `git checkout`/`reset`/`clean` and is never committed (keep it out of every commit; it is local signing config).

## Per-change delivery workflow (STRICT — the user tests between every step)

The user builds, sideloads and tests **every** change on-device before it is ever committed. Follow this ordering exactly, same as the AppManager / Jami / ArcaneChat forks:

1. **Deliver the patch + the apply-and-build block ONLY.** Generate the change as a `.patch` (per `patch-naming`) against `origin/custom`, present it, and give the shell block that applies it and builds/signs/sideloads. **Then stop.** Do **NOT** include any `git commit`, `git push`, `git fetch`, rebase, or `git reset --hard origin/custom` in that turn — not even "for when you're ready". Pre-empting the push is wrong every time.

   **Always paste the COMPLETE build block, every time** — including on a re-send after a fix. Never tell the user to "change only the `patch=` line" or "re-run the same block with X edited"; they copy-paste the whole thing in one go and a partial instruction invites a stale/mismatched run. Each delivery is the full, self-contained apply-and-build block with the correct `patch=` filename already filled in.
2. **Wait for the user to test.** They may report problems; iterate with more build-only patches. Fix patches can stack on the uncommitted working tree (the build block stays `git checkout custom`, no reset, so prior uncommitted patches stay in place).
3. **Only when the user explicitly says "Push"** do you provide, in that turn: the commit + push block (stage explicit source paths — never `keystore.properties`, never the submodules, never `build/`; `git commit`; `git push origin custom`), and then re-sync the sandbox on the user's next "Done". **The push block has a SINGLE y/n gate — confirm-before-commit — after which it commits and pushes in one go.** Do NOT add a second "Push to origin? (y/n)" prompt; the user had it removed (the staged-summary + `git status --short` shown before the one gate is the review point).

One logical change per patch; build-only until the user says "Push". If several related uncommitted patches were tested together, they may be squashed into one commit at push time when that reads as one logical change. New feature commits append on top of commits 1–2 and survive rebases.

### Sandbox pre-patch protocol (Claude's side)

Before generating any patch, sync the sandbox clone to `origin/custom` as it exists on GitHub right now (the fork's `origin` is on github.com, reachable from the sandbox; submodules are not needed to diff tracked source):

```bash
cd /home/claude/shiroikuma-futokxkb   # or wherever the sandbox clone lives

git reset --hard origin/custom

```

Generate the patch against `origin/custom`. Never assume local sandbox state matches; never base a patch on a stale commit or a tag. Working from drift is what produces "patch does not apply" after the user reports a successful push.

### User-side apply guard

Before every `git apply`, reset to a known base and clear leftover untracked files from prior patches (a `git reset --hard` reverts only *tracked* files; an untracked file a previous patch created survives and makes the next `git apply` abort the **entire** patch atomically with `already exists in working directory`). Scope the clean so it does not nuke `build/` or the `java/assets/layouts` submodule:

```bash
r git reset --hard origin/custom      # or: git checkout custom, to keep stacked uncommitted fix patches
# Enumerate the source-bearing dirs explicitly with -fdx. Do NOT use a bare `git clean -fdx java/`:
# the `-x` would wipe the java/assets/layouts submodule contents and any build/ outputs under java/.
# Every product-flavor source root MUST be listed (java/stable, java/unstable, java/playstore), not
# just java/src — an abandoned experiment that wrote e.g. java/stable/res/ is untracked, so a reset
# leaves it behind, and if it's outside the clean scope it survives into later builds/commits and
# the artifact guard (correctly) blocks the push. (This bit us once: a dropped stable-flavor label
# override left java/stable/res/ lingering.)
r git clean -ndx java/src common/src java/res java/assets-custom java/stable java/unstable java/playstore
r git clean -fdx java/src common/src java/res java/assets-custom java/stable java/unstable java/playstore
r git apply ~/tmp/<patch-file>.patch
# post-apply sentinel grep (r does NOT abort the block on a failed apply):
r grep -n "<a string the patch introduces>" <file the patch touches>
```

## Live look/size knobs (Multiling-style) — LIVE (core set shipped)

A Multiling-O-style per-geometry sizing screen (`KxkbSizingScreen`) with **live sliders + colour pickers** — the docked keyboard updates as the user drags. Replaces hardcoding look values on the `HighContrastYellow` theme. Shipped incrementally `c94a849` (look fields) → `195ccf075` (split snap) → `23ae74876` (per-geometry colours + geometry-overlay sync) → `9a143522c` (cluster main-position sliders + the screen rename "Live sizing" → "Keyboard UI"); see "Implemented features" for the per-commit chain. This section is the architecture map for maintenance / rebases / adding more knobs.

**Settled design decisions (kept):**
- **Per geometry**, keyed off FUTO's `KeyboardSizeSettingKind` — `Portrait`, `Landscape`, `FoldableInnerDisplay` (the Mate XT's 6 fold×orientation states collapse into these). `LastUsedSizeStateSetting` auto-tracks the active geometry.
- **One unified slider screen**, every knob live.

**The backbone is FUTO's per-kind store, now keyed per-combo.** `SavedKeyboardSizingSettings` (`@Serializable data class`, `KeyboardSizingCalculator.kt`) holds the whole blob (sizing + look + colours). Originally persisted **per geometry**: `KeyboardSettings[kind]` maps each `KeyboardSizeSettingKind` to a `SettingsKey` holding the JSON blob. **As of `7b399be0b` it is per-(active subtype × geometry):** a shared top-level `perComboSizingKey(context, kind)` folds the active subtype (the language+layout pair, via `Subtypes.subtypeToString(getActiveSubtype(context))`, sanitised) into a dynamic key `keyboard_settings__<subtype>__<kind>`; it returns `null` when no subtype resolves (BFU / direct-boot) so callers fall back to the geometry key. `getSavedSettings()` reads per-combo → **inherits** the geometry blob → `getDefaultSettingForKind`; `editSavedSettings { it.copy(...) }` reads that same effective value but **writes the per-combo key**, so an unedited combo shows the geometry baseline and **forks to its own blob on first edit** (the three original geometry blobs stay as the read-only inherited baseline — automatic migration, nothing reset). `resetCurrentMode`/`exitOneHandedMode` inherit this for free (they route through `editSavedSettings`). `fromJsonString` is lenient (`ignoreUnknownKeys`), so new fields with defaults deserialize cleanly on old blobs. **The exact same `perComboSizingKey` is used by the calculator, the screen, and the live watcher — never reconstruct the key string inline anywhere, or the three drift and edits silently miss.** That struct is the single source of truth the screen edits.

**Fields on `SavedKeyboardSizingSettings`** (all per-geometry):
- *Sizing* (Phase 0b, `89b727d`): additive `horizontalGapAddDp` / `verticalGapAddDp` (neutral `0`) + top/bottom row-height factors; the existing `heightMultiplier` / `paddingDp` / `splitWidthFraction` / `prefersSplit` are reused for keyboard height / position / split — no new position fields.
- *Look* (Phase 0a, `c94a849`, + hint later): `fontSizeMultiplier`, `labelWeight`, `keyRoundness`, `borderWidthDp`, `hintSizeMultiplier` — each defaults to sentinel **`-1` = inherit the theme**.
- *Colours* (`23ae74876`): 8 nullable packed-ARGB `Int?` (`null` = inherit) — `fontColor`, `secondaryFontColor`, `keyBackgroundColor`, `functionalKeyBackgroundColor`, `keyBorderColor`, `keyboardBackgroundColor`, `suggestionBarColor`, `suggestionTextColor`.

**Two render paths, both live:**

1. **Sizing** rides the per-kind calc rails in `LayoutEngine.kt` / `KeyboardSizingCalculator.kt`:
   - *Gaps* — additive dp: `gap + horizontalGapAddDp.dp` (neutral 0, so it works on the zero-gap HCY theme).
   - *Row heights* — `heightMultiplier` drives overall height (`singularRowHeight`). Top/bottom factors are a **redistribution within the fixed total**: `topRowIndex = rows.indexOfFirst { !isBottomRow }`, the top row is scaled `rowHeight*rowHeightPx*topF` with `topRowExtraWeight` folded into the row-height denominator; the bottom row is pinned `standardRowHeight*bottomF` (Fixed mode). **Critical:** the stock `clampHeight` (ClampHeight mode) pins keys to `standardRowHeight` and turns surplus into vertical gap → "phantom gaps" when resizing; gated off via `liveRowSizingActive = topF != 1.0 || bottomF != 1.0` so keys fill their cells. `topRowHeightFactor` threaded LayoutParams → KeyboardLayoutSetV2Params → `KeyboardSwitcher.java`.
   - *Suggestion-bar height* — the hardcoded `ActionBar.kt val ActionBarHeight = 40.dp` became `LocalActionBarHeight = compositionLocalOf { ActionBarHeight }`, provided at the keyboard root (`UixManager.ProvidersAndWrapper`) from `actionBarHeightState` + `refreshActionBarHeight()` (reads `calculateSuggestionBarHeightDp() = 40*factor`). All composable read sites repointed to `LocalActionBarHeight.current`; non-composable sites read `sizeCalculator.calculateSuggestionBarHeightDp()`. Insets (`calculateTotalActionBarHeightPx`) already matched. `refreshActionBarHeight()` runs in `onSizeUpdated` and the collector.

2. **Look** is overlaid per-build in `LatinIME.withPerKindLook(scheme)`: reads the current geometry's blob and `.copy()`s the `>= 0` fields onto `scheme.extended.advancedThemeOptions` — `keyLetterScale` ← fontSizeMultiplier, `keyLabelWeight` ← labelWeight, `keyRoundness` ← keyRoundness, `keyStrokeWidthDp` ← borderWidthDp, `keyHintScale` ← hintSizeMultiplier. `activeColorScheme` holds the **clean base** scheme; the overlay only enters the provider, so inherit stays correct. Applied at both `BasicThemeProvider` construction sites (`updateDrawableProvider` + the `getDrawableProvider` fallback). Draw sites: main label `* getKeyLetterScale()` (`KeyboardView.java:449`, auto-fit to key width); roundness `keyCornerRadius = 9.dp*roundness` (space `48.dp*roundness`, pill-clamped, `BasicThemeProvider`); **hint scale needed NEW plumbing** — `AdvancedThemeOptions.keyHintScale` → `DynamicThemeProvider`/`BasicThemeProvider` → `* getKeyHintScale()` at the on-key hint draw sites `KeyboardView.java:499 & 537` (long-press popup hint unchanged).

3. **Colours** are overlaid in the SAME `withPerKindLook(scheme)` pass (`23ae74876`): each non-null colour `.copy()`s onto `scheme.base.outline` (key border) or the **extended** scheme — `onKeyboardContainer` ← fontColor, `hintColor` ← secondaryFontColor, `keyboardContainer` ← keyBackgroundColor, `keyboardContainerVariant` ← functionalKeyBackgroundColor, `keyboardSurface`(+`keyboardSurfaceDim`, and clear `keyboardBackgroundGradient`) ← keyboardBackgroundColor. The suggestion bar gets **two dedicated `ExtraColors` fields** (`ColorScheme.kt`): `suggestionBarColorOverride` + `suggestionTextColorOverride` — separate because the bar bg otherwise shares `keyboardSurface` with the keyboard bg, and suggestion text shares `onSurface` with other chrome; `ActionBar.kt` `actionBarColor()`, the suggestion-word colour (~304) and CJK `CandidateItem` colour (~1048) prefer them when set. Compose `Color` import in `LatinIME` is **aliased `as ComposeColor`** (clashes with the existing `android.graphics.Color` used for `Color.BLACK`/`TRANSPARENT`). `lookFieldsDiffer` includes the 8 colour fields. The **Compose chrome (suggestion bar) reads the overlaid scheme**, not the clean one: `LatinIME.overlaidColorScheme` (an `activeOverlaidScheme` mutableState set inside `updateDrawableProvider`/`getDrawableProvider`) is what `UixManager` feeds to `UixThemeWrapper`, so bar/suggestion colours recompose live.

**Live-refresh wiring** (`LatinIME.kt`, the "Listen to size/look changes" `dataStore.data.collect`). Since `7b399be0b` the watcher follows the **dynamic per-combo key** (`perComboSizingKey(this, currentSizeState)`, falling back to the geometry key), NOT the three fixed geometry keys. Each emission it diffs that key + its blob against the previous: **same key + new blob** = a live edit → `invalidateKeyboard(refreshSettings = true)` (recomputes `size.value = calculateSize()` AND `recreateKeyboard()`); **key changed** = a combo switch (language/layout, or geometry after a write) → force `updateDrawableProvider` + `invalidateKeyboard`. A `lookFieldsDiffer(old, new)` check (look, colour, **and** the flick/cluster draw-offset fields) gates whether `updateDrawableProvider(activeColorScheme.value)` also runs so look/colour/offset-only edits repaint without a full rebuild (a key change always forces it). **⚠ TWO gotchas, both have bitten:** (1) **Any NEW per-geometry look/offset field MUST be added to `lookFieldsDiffer`** — it's the gate that decides whether `withPerKindLook` re-runs (the only thing that re-pushes the static offsets to the draw path); omit it and a change to that field alone updates the stored blob but never the screen (first happened with `clusterLeftOffset`/`clusterRightOffset`). (2) **The watcher MUST key off `perComboSizingKey`, not `KeyboardSettings[kind]`** — when the per-combo store landed, leaving the watcher on the geometry keys made every live slider/colour edit a no-op (the edit wrote the per-combo key the watcher wasn't watching; only a geometry rebuild, which re-reads the store directly, showed it). Driving combo switches from this watcher (rather than `onNewSubtype`) is deliberate: `DataStoreHelper`'s cache (read by `getSettingBlocking` → `perComboSizingKey` → `getActiveSubtype`) is updated by a background `dataStore.data` collector, so it trails the datastore by up to one emission; the watcher converges on whichever emission the key actually flips, whereas a cache read inside `onNewSubtype` can fire a beat early on the *old* combo. A secondary `lastProviderSubtype` gate in `recreateKeyboard` (parallel to `lastProviderSizeState`) is kept as a safety net so any recreate path rebuilds the overlay when the subtype changed.

**⚠ Per-geometry GEOMETRY-CHANGE wiring — the overlay is baked into the provider, so a rotate/fold must rebuild it (this bit us hard, `23ae74876`):** the `withPerKindLook` overlay (fonts, roundness, border AND the 8 colours) is baked into the `BasicThemeProvider` + the overlaid chrome scheme at build time. It is otherwise rebuilt only on a look-field/colour edit (collector) or theme change — **never on a pure geometry switch** — so rotating/folding kept the *previous* geometry's look+colours until the next edit (the "stuck, then jumps" symptom). Two fixes, both required:
   - **`onConfigurationChanged` must call `onSizeUpdated()`.** It's the one geometry-changing event that did NOT flow through `onSizeUpdated`, so without it `publishCurrentSizeState()` never fired on rotation → `LastUsedSizeStateSetting` stayed pinned → the settings screen's `kind` stayed on the old geometry → **edits wrote to the wrong geometry's blob** (set in landscape, appeared in portrait, lost in landscape). (Fold already republishes: the `WindowInfoTracker` collector in `UixManager` calls `invalidateKeyboard` → `publishCurrentSizeState`.)
   - **The provider must be re-synced to the live geometry at the recreate chokepoint.** `recreateKeyboard()` now rebuilds the overlay (`if (currentSizeState.name != lastProviderSizeState) updateDrawableProvider(activeColorScheme.value)`) before loading — every recreate path (rotation via `onSizeUpdated`→`invalidateKeyboard`, fold via collector→`invalidateKeyboard`) flows through it. `updateDrawableProvider` records `lastProviderSizeState`. `onSizeUpdated` also ORs `geometryChanged = currentSizeState.name != lastProviderSizeState` into `shouldInvalidateKeyboard` so a recreate is forced even if the two geometries happen to share dimensions. **Why not just rebuild in `publishCurrentSizeState`:** the orientation flag can lag the view resize a beat, so a rebuild there reads the geometry you're *leaving* (off-by-one). Keying off `lastProviderSizeState` at recreate time is timing-proof.

**Split model** (`SplitKeyboardSize` in `KeyboardSizingCalculator.kt`; positioning in `LayoutEngine.kt`):
- `splitLayoutWidth = (displayWidth * splitWidthFraction).coerceInLoosely(dp(48), displayWidth)` — cap is the **full display width** (was `9/10`), so the halves can fully close the centre gap and overlap the duplicated middle column.
- Splittable rows (`splittable = letters != null` → letter/number rows; **the bottom row is non-split**) → `splitRow()` duplicates the middle column on odd layouts → `addRowAlignLeft` (left half flush x=0) + `addRowAlignRight` (right half flush right); centre gap = `displayWidth − leftWidth − rightWidth`.
- **Snap to one column** (`195ccf075`): `splitShouldMerge = isSplitLayout && (layoutWidth + regularKeyWidth > displayWidth)` — past the "doubled-column-adjacent" point (~`maxCols/(maxCols+1)` of the display). When set, the splittable row is rendered **un-split and centred** (`addRow(idx, row.entries, offset, …)`, `offset = (displayWidth − layoutWidth)/2`); un-splitting drops the duplicated column, so the two halves collapse to one continuous column — skipping the partial-offset zone between adjacent and merged.
- **Bottom (action) row** is laid out at the **constant full width** (`unsplitLayoutWidth`), so ALL its key-width bounds must derive from that width or the keys deform as the split narrows. Three repointed to `unsplitLayoutWidth`: `bottomRegularKeyWidth = 0.1f * unsplitLayoutWidth`, `minimumBottomFunctionalKeyWidth = unsplitLayoutWidth * minimumBottomRowFunctionalKeyWidth`, and the shared (split-capped) `functionalWidth` is **rescaled** `functionalWidth * unsplitLayoutWidth / layoutWidth` for the bottom row (shift/delete take the shared functional width, otherwise capped at `layoutWidth*0.15`). All no-ops at 100% split and for non-split layouts. (The user's `kxkb.yaml` sets `minimumFunctionalKeyWidth: 0.0` + `minimumBottomRowFunctionalKeyWidth: 0.0` so functional keys render at regular width, and the spacebars are `Custom1: 0.2` — **not** Grow — which is why the slack surfaced as deformation rather than being absorbed.)

**The screen** — `KxkbSizingScreen` + `KxkbSizingResetMenu` (`Typing.kt`): reuses the stock `SettingSliderForDataStoreItem` with a **synthesized `DataStoreItem`** (`floatItem`) per knob. **As of `7b399be0b` it binds to the per-combo key:** it reads `ActiveSubtype` reactively (`useDataStore`) and computes `perComboSizingKey(context, kind)` in a `remember(activeSub, kind)` (so the sliders rebind when you switch layout/language with the screen open — `useDataStore` reads live from the cache by key, so a changing key just works), reads per-combo → geometry-blob → default for display, and writes the per-combo key via `comboItem.setValue` (forking on first edit). A "language · layout" scope line (built from `Subtypes.getName` + `getLayoutName(SubtypeLocaleUtils.getKeyboardLayoutSetName(st))`) sits above the geometry line so the edited combo is visible. `defaultSizing = getDefaultSettingForKind(kind, context)` supplies the inherit/neutral defaults the reset restores. The top-of-page explainer (`kxkb_sizing_geometry_note`) still notes the geometry scope. **Colours** are 8 expandable `ColorSetting` rows (a private composable + `ColorChannelSlider`); since `4bf0c99af` each sits in the subgroup of the element it affects (Jami-style) rather than in one "Colours" section: each row = a live swatch + an "Inherited" tag (when `null`) that expands to Alpha/Red/Green/Blue channel sliders (reusing `SettingSliderForDataStoreItem`, 0–255), seeded from the active theme via `LocalKeyboardScheme.current` (`.toArgb()`), writing the packed ARGB to the blob via a `writeColor { it.copy(...) }` helper. Reset clears the 8 colours to `null` too. Strings in `java/res/values/strings-uix.xml`. **Page structure (`4bf0c99af`):** controls are grouped by two private header composables — `KxkbSection(title)` (large `Typography.Heading.MediumMl` accent title + full-width 2dp accent `HorizontalDivider`) and `KxkbSubgroup(title) { content }` (accent title with a short underline sized to the text via `Modifier.width(IntrinsicSize.Min)` + `softWrap=false`, then its controls in an indented `Column(padding(start = 32.dp))`). Indents cascade: section start 16dp < subgroup header 24dp < controls 32dp (+ the control's own ~12–16dp). Sections: Size (Height / Spacing / Split mode) · Keys (Text / Shape & background / Hint labels) · Sliding labels (Flick / Cluster positions) · Surfaces & suggestion bar (Keyboard background / Suggestion bar) · Layout switcher. Section/subgroup titles are English literals (no string churn). **To add a knob:** drop a `SettingSliderForDataStoreItem`/`ColorSetting` inside the right `KxkbSubgroup { }` lambda — it inherits the indent automatically; don't place it outside a subgroup.

**Current live knob set** (per-combo since `7b399be0b`; was per-geometry): horizontal gap (0–10dp), vertical gap (0–10dp), keyboard height (`heightMultiplier` 0.5–2.5×), top row (0.5–2.0× redistribute), bottom row (0.2–2.0× redistribute), key font size (0.5–3.0×, inherit→1.4 on HCY), secondary/hint text size (0.5–3.0×, inherit→1.0), key roundness (0–5, inherit→0 on HCY), suggestion-bar height (0.5–2.0×), split width (40–100%), **8 colours** (key font, secondary/hint font, key bg, functional-key bg, key border, keyboard bg, suggestion-bar bg, suggestion font — each RGBA, inherit-by-default), + Reset live sizing.

**Phase 4b-2 — compact slot shorthands (SHIPPED `3b59ae18b`).** The eight compass direction fields changed from `Key?` to `Slot?`, where `typealias Slot = @Serializable(with = CompassSlotSerializer::class) CompassSlot` (mirrors FUTO's `typealias Key = @Serializable(with = KeyPathSerializer::class) AbstractKey`). `CompassSlot` is a plain `sealed interface` (NOT `@Serializable` — the custom serializer handles it) with `KeySlot(key: Key)` and `ModSlot(mod, on?, label?)`. `object CompassSlotSerializer` peeks at `(decoder.beginStructure(descriptor) as YamlInput).node` (the `ClassOrScalarsSerializer` pattern): if it's a `YamlMap` whose keys (`node.entries.keys.map { it.content }`) **lack `type`** and contain `mod`/`chord`/`macro`, it decodes a private `@Serializable data class` (`ModShorthand`/`ChordShorthand`/`MacroShorthand`) → `ModSlot` / `KeySlot(ChordKey(keys))` / `KeySlot(MacroKey(text))`; otherwise it falls back to `KeySlot(decodeFromYamlNode(KeyPathSerializer, node))` (scalar, list, or `type:`-tagged map = a normal key). chord/macro desugar immediately (they don't need the primary); **`mod` defers** to a `ModSlot` because it needs the primary's character, which only `CompassKey` knows. `CompassKey.computeData` now extracts `primaryChar` from the primary's `ComputedKeyData` (`code > 0 → String(Character.toChars(code))`, else its `label`) and `resolveSlots(primaryChar)` converts each `Slot` via `slotToKey`: `KeySlot → key`; `ModSlot → ChordKey(keys = modPrefix(mod) + "-" + (on ?: primaryChar), label)`. `modPrefix`: ctrl/control→`C`, alt/meta/opt/option→`M`, shift→`S`, super/win/cmd/gui→`s` (also accepts the single letters). So on primary `o`: `{ mod: ctrl }` = `C-o`, `{ mod: alt }` = `M-o`, `{ mod: ctrl, on: x }` = `C-x`; plus `{ chord: "C-x C-s" }` and `{ macro: "etc. " }`, all taking an optional `label`. **GOTCHA (now fixed by this commit):** before 4b-2 a typeless inline map in a slot failed kaml with *"Property 'type' is required"*; the shorthands now resolve that — but a standalone (non-slot) special key still needs the full `type:` form (the shorthand serializer is only on compass slots). **Phase 4c — `cycle` (multitap) key (SHIPPED `05388366e`).** New `CycleKey` (`@SerialName("cycle")`) in `CompassKey.kt`. Repeated taps cycle through a set, each tap deleting the previously-committed entry and committing the next (wrapping); the multitap window lapsing, the cursor moving, or any other key ends the cycle so the next tap starts fresh. **`taps`** is a string (one tap per codepoint) OR a list (multi-character entries), via `@Serializable(with = TapsSerializer::class) data class Taps(items: List<String>)` whose serializer peeks the node (`YamlScalar` → `content.codePoints()` each → one tap; `YamlList` → each entry a tap). **Carrier:** like macro/chord the tap-set rides in `outputText`, here behind a leading **`\u0001` sentinel with entries joined by `\u0001`** (both impossible in real text). `InputLogic.onTextInput` intercepts `startsWith("\u0001")` (beside the chord `\u0000` check) → `performCycle`, which keeps **state** (`mCycleTaps`/`mCycleIndex`/`mCycleTime`/`mCycleAnchor`): a tap continues only if the SAME tap-set is re-tapped within the window AND the caret is still where the last commit left it (`getExpectedSelectionStart() == mCycleAnchor`, start==end); else fresh. Continuing deletes `mCycleTaps[mCycleIndex].length()` UTF-16 units (`deleteTextBeforeCursor` → `deleteSurroundingText(n,0)`, so `.length()` is the right count even for astral/multi-char entries) then advances `(index+1)%n` and `commitText`s. Commits directly via the connection (composer flushed via `commitTyped` first); **no composer/suggestions involvement, no new code constant, no listener/PointerTracker change.** **The multitap window is a SETTING:** `Settings.PREF_MULTITAP_TIMEOUT = "kxkb_multitap_timeout"` + `readMultitapTimeout(prefs)` (default 800ms) → `SettingsValues.mMultitapTimeout` (read in `loadSettings`, so it re-applies live on pref change) → read by `performCycle`. Exposed as a `SettingSliderSharedPrefsInt(key = "kxkb_multitap_timeout", default = 800, range 300..1500ms)` in a new **Cycle section on the Special Keys page**. 5 files: `CompassKey.kt`, `InputLogic.java`, `Settings.java`, `SettingsValues.java`, `SpecialKeys.kt`. **Invalidation is implicit** (caret-anchor + timeout) — no hot-path reset hook; the only false-resume is manually moving the caret back to the exact anchor within the window, transient and harmless.

**Phase 4d — `cluster` (predictive multi-key) (SHIPPED `16d9b57a1`).** New `ClusterKey` (`@SerialName("cluster")`) in `CompassKey.kt` — the old Multiling `3+2`. A wide key carries a band of N **main** glyphs (`main: "aev"`, left→right); a plain tap commits the **centre** main, but the real touch position is decomposed across ALL the band's letters so the language model picks one from word context. Unlike chord/macro/cycle it does **NOT** intercept `onTextInput` — the tap is an ordinary centre-main keypress; the predictive behaviour is entirely a spatial-model injection. **A cluster IS a compass whose middle row is the band:** the desugar builds a compass-shaped `ComputedKeyData` with `code` = centre main (= `main[n/2]`), flick `West` = first main / `East` = last main (precise side access by a slide), and the six diagonal/vertical slots = "extras" (any key or `{ mod }`/`{ chord }`/`{ macro }` shorthand — the SAME `Slot`/`CompassSlotSerializer` machinery as compass). `left`/`right` are reserved (the side mains); only `up`/`down`/`upLeft`/`upRight`/`downLeft`/`downRight` are author-settable. It tags the `ComputedKeyData` with **`clusterMains: List<ClusterMain>`** (`ClusterMain(codePoint, column, columnCount)` — defined in `KeyData.kt`), carried onto `keyboard.Key` in `computedKeyToKey`.

**The prediction hook (the whole point) is pure Java in `ProximityInfo.createNativeProximityInfo` — NO native/C++ change.** FUTO's transformer decoder (on by default, `PREF_KEY_USE_TRANSFORMER_LM`) disambiguates each tap via the native `proximity_info.h::decomposeTapPosition(x,y)`, which is purely geometric: it overlaps a tap-radius circle (`MOST_COMMON_KEY_WIDTH*0.292`) with every key's rect and weights `getKeyCodePoint(key)` by the overlap, feeding the top `NUM_TOKEN_MIX` (=4) letters as a per-position distribution to `PredictCorrection` (the literal typed char is just an a–z gate). Those native key arrays (`keyXCoordinates/keyCharCodes/…`) are **built in Java and the native side consumes whatever list it's handed** — so `clusterSubKeys(key)` emits **one tiled sub-rect entry per LETTER main** (column `c` of `N` → `x = keyX + c*keyW/N`, full key height, code = that main) INSTEAD of the key's single full-width entry, wired into `getProximityInfoKeysCount`, the coordinate-array fill, AND the sweet-spot loop in lockstep (so the arrays stay length-aligned with `keyCount`). A tap then decomposes across the band: centre weighted highest (tap centred on it), sides live, and leaning left/right nudges the weights — exactly Multiling. **Letter-mains-only:** non-letter mains (`(`, `;`, digits) still commit (centre-tap or slide) but get no proximity entry — the transformer ignores non-letters anyway, and skipping them keeps the entry count under `MAX_KEY_COUNT_IN_A_KEYBOARD = 64` (the hard native cap; mind it for big cluster layouts). The real `(x,y)` reaches the composer only because `hasProximityCharsCorrection(code)` is true — v2 layouts set `mProximityCharsCorrectionEnabled = true` unconditionally and the gate also passes for any letter / the whole alphabet element; on a non-letter-only path it would send `NOT_A_COORDINATE` and the spatial decomposition would not run.

## Swipe-space layout switcher (Multiling-style, COMPLETE — phases 1–3 shipped, latest `dfcd1002a`)

A Multiling-style fast switcher: swipe off the spacebar to open a panel covering the keyboard, slide the finger onto a cell, release to switch — all in one motion. It is a 3-column grid: left = settings shortcuts, middle = other languages, right = current-language layouts, each column with a header on a shared top row ("Keyboard UI" → the sizing screen, "Keyboards…" → the system IME picker, "Custom Layouts" → the Custom-layouts editor). The panel is **compact** (anchored near the swipe, fitted to the keyboard width), not full-width, so short reaches work in the wide unfolded geometries. Phasing (all done): (1) gesture + overlay + the **layouts column** `796423b8c`; (2) the **languages** column + **recency tracking** `2e08b1cbf`; (3) the **header row** + **configurable left-column shortcuts** `dfcd1002a`. This section is the architecture map.

**The model that makes the single-motion work — `MoreKeysPanel` as an overlay.** A Compose dialog (like `showLanguageSwitcher`) can't do single-motion because it's a separate window that wouldn't receive the *continuing* spacebar touch. FUTO's own more-keys-panel infra already does "open mid-gesture → track finger → release-to-select", so `LayoutSwitcherView` (`…/keyboard/LayoutSwitcherView.java`, NEW) implements `MoreKeysPanel` and `PointerTracker` drives it for free: once `mMoreKeysPanel` is set, `PointerTracker.onMoveEvent`/`onUpEventInternal` translate the touch and forward `onMoveEvent`/`onUpEvent` to the panel, then dismiss after up. To open it, `PointerTracker.openLayoutSwitcherPanel()` mirrors the long-press `showMoreKeysKeyboardOnPointer` path: `sDrawingProxy.showLayoutSwitcher(this)` (new `DrawingProxy` method, impl in `MainKeyboardView`) builds + shows the view, then `onDownEvent` + assign `mMoreKeysPanel`.

**The identity-coordinate trick (the bit that's easy to get wrong otherwise).** `PointerTracker` forwards keyboard-local coords. Rather than do the `MoreKeysKeyboardView` container/window-origin/`getDefaultCoordX` translation dance, `LayoutSwitcherView` is sized to **exactly cover the MainKeyboardView** and positioned at its window origin inside `mDrawingPreviewPlacerView` (via `showInParent`, `setTranslationX/Y(mWinX/mWinY)` from `parentView.getLocationInWindow`). So keyboard-local == panel-local and `translateX/Y` are the **identity**. All drawing + hit-testing happen in keyboard-local coords; the view renders the **N columns side by side** (left shortcuts, middle languages, right layouts). Each column is sized to its own longest label but **every column is capped to an equal `avail/nCols` share so all three always fit the keyboard width** (text is `clipRect`-clipped per cell so a long label can't overflow), and the block is centred on the swipe x (clamped on-screen). Within a column, entries stack **upward** from a baseline just above the finger (index 0 = bottom = nearest, so the most-recent is closest); the **header sits on a single shared top row** across all columns (row index = max entry count), so headers line up even when columns differ in length. A dim scrim (`0xB0000000`) backs it; colours come from the provider (`keyboardColor`/`keyColor`/`moreKeysTextColor`), the highlighted cell inverts (text-colour bg + keyboard-colour text), **header cells fill with `keyboardColor`** to read as section tops vs the `keyColor` entry cells, and the active subtype is marked with a bullet. `onShowMoreKeysPanel` must run **after** `mKbWidth/mKbHeight` are set (they feed the `showInParent` LayoutParams).

**The trigger — a new spacebar mode.** `Settings.SPACEBAR_MODE_SWIPE_LAYOUT_MENU = 3` (alongside the existing 0/1/2; picker + label in `Typing.kt` / `strings-uix.xml`). In `PointerTracker`, the spacebar sliding branch (the `mIsSlidingCursor && CODE_SPACE` block in `onMoveEventInternal`): when mode == LAYOUT_MENU and `!mSpacebarLongPressed`, a real swipe (`abs(x-mStartX) > sPointerStep` past the swipe-ignore time) calls `openLayoutSwitcherPanel()` and returns (no cursor move); once the panel is open, `onMoveEvent`'s `isShowingMoreKeysPanel()` branch takes over so this code isn't re-reached. **Long-press → cursor**: LAYOUT_MENU is added to the same `onLongPressed` branch as `SWIPE_LANGUAGE` (sets `mSpacebarLongPressed`, `onMovingCursorLockEvent(true)`), after which the sliding branch falls through to `onMovePointer`. Net: swipe → menu, long-press+drag → cursor, exactly as chosen.

**Subtype model (how cells map to switches).** A FUTO subtype IS a `(language, layout)` pair stored as `locale:KeyboardLayoutSet=<layout>:<langtag>`; the enabled `SubtypesSetting` list is the user's languages-and-layouts, `ActiveSubtype` is current. Every cell (layout OR language) carries a target subtype string; selecting it just calls `Subtypes.switchToSubtypeString` → writes `ActiveSubtype` → `LatinIME` observes it and performs the actual switch. Helpers in `Subtypes.kt`: `getActiveSubtypeString`; `getCurrentLanguageLayouts` (enabled subtypes whose language matches the active one — `langOf(sub)` = `getLocale(sub.split(":")[0]).language` — ordered **active-first → recency → enabled**, capped 4, as `(subtypeString, getLayoutName)` pairs) for the layouts column; and `getOtherLanguageEntries` for the languages column (distinct **other** languages among enabled subtypes, ordered recency-then-enabled, capped 4; each entry targets that language's **last-used layout** — first recency hit for the language, else its first enabled subtype — labelled with the native name via `getLocaleDisplayName(locale, locale)`). Since phase 3 the panel **always opens** (the headers + at least one shortcut are always present), so there is no longer a no-switch no-op guard.

**Recency tracking (phase 2).** `LayoutRecency = SettingsKey(stringPreferencesKey("kxkb_layout_recency"), "")` stores recently-active subtype strings newline-joined, most-recent first (a subtype string can't contain `\n`). It is updated from the **central** `LatinIME` `ActiveSubtype` observer (`onNewSubtype`, the `dataStore.data.collect`), so EVERY switch is captured — panel, language dialog, swipe-cycle, settings — not just panel switches. `Subtypes.pushLayoutRecency` is `suspend` (called from that observer coroutine; reads via `getSetting`, writes via the `suspend` `setSetting` — no `runBlocking` nesting), prepends + dedups + caps at 16. **No recursion:** it writes `LayoutRecency`, not `ActiveSubtype`, and `currentSubtype` is already updated, so the re-triggered collect is a no-op. Before any recency accumulates, both helpers fall back to enabled order, so it works from first launch.

**Command cells + the header row (phase 3).** A cell's target is now either a subtype string (switch, as above) **or a command**, dispatched in the panel's `onUpEvent` → `activate(target)`: `!nav/<route>` builds an `Intent` to `SettingsActivity` with `putExtra("navDest", route)` (+ `NEW_TASK|SINGLE_TOP|CLEAR_TOP`) and `startActivity` — exactly the `CustomLayoutsAction` pattern; `!ime` calls `((InputMethodManager) getSystemService(INPUT_METHOD_SERVICE)).showInputMethodPicker()`. Routes are the registered Compose-nav routes in `SettingsNavigator.kt` (`kxkbSizing`, `languages`, `addLanguage`, `themes`, `specialKeys`, `resize`, `home`, and **`devlayouteditor`** = the Custom-layouts editor — *not* `devlayouts`). `setColumns` keeps the same `(target, label)` Pair shape (commands are just special targets) but gained a parallel `List<Pair> headers` (one per column, nullable) + an `isHeader` flag on `Cell`. The three headers: left `!nav/kxkbSizing` "Keyboard UI", middle `!ime` "Keyboards…", right `!nav/devlayouteditor` "Custom Layouts" (renamed from "Layouts…" in `b51800df5`).

**Configurable left-column shortcuts (phase 3).** The left column is no longer hardcoded — it's built from a central catalog in `Subtypes.kt`: top-level `data class SwitcherShortcut(id, label, target, defaultOn)` + `val LayoutSwitcherShortcutCatalog` (add_layout/custom_layouts/themes/special_keys/resize/languages/settings; add_layout+themes+settings default-on) + `fun layoutSwitcherShortcutPrefKey(id) = "kxkb_switcher_shortcut_$id"`. `Subtypes.getLayoutSwitcherShortcuts(context)` filters the catalog by a **SharedPreferences boolean per id** and maps to `(target, label)`; `MainKeyboardView.showLayoutSwitcher` calls it for the left column. The toggle UI is a section ("Layout switcher shortcuts") at the bottom of `KxkbSizingScreen` ("Keyboard UI"), one `SettingToggleSharedPrefs(title, key=layoutSwitcherShortcutPrefKey(id), default=defaultOn)` per catalog entry. **Store choice is deliberate:** `SettingToggleSharedPrefs`→`useSharedPrefsBool` and the panel's `PreferenceUtils.getDefaultSharedPreferences(context).getBoolean(...)` hit the *same* default SharedPreferences (the proven key-sliding round-trip), so UI and panel agree without a DataStore hop — don't "fix" this to a `SettingsKey`/DataStore toggle or the two stores diverge. Changes apply next time the panel opens (no rebuild). Labels are English literals (consistent with the hardcoded header labels); localise later if wanted.

**Key files:** `…/keyboard/LayoutSwitcherView.java` (the panel — multi-column via `setColumns(columns, headers, …)`, the `Cell` model + `isHeader`, fit-to-width layout with the shared header row, and `activate()` command dispatch — imports `Intent`, `InputMethodManager`, `SettingsActivity`), `…/keyboard/PointerTracker.java` (`openLayoutSwitcherPanel` + the swipe hook + the LAYOUT_MENU long-press branch), `…/keyboard/MainKeyboardView.java` (`showLayoutSwitcher` — builds the 3 columns + 3 headers, left column from `getLayoutSwitcherShortcuts`, pulls theme colours, anchors at the last touch), `…/keyboard/internal/DrawingProxy.java` (the `showLayoutSwitcher` method — only `MainKeyboardView` implements `DrawingProxy`, so no other impl to update), `…/latin/Subtypes.kt` (`LayoutRecency` key + `getLayoutRecency`/`pushLayoutRecency`; the switcher helpers `getCurrentLanguageLayouts`/`getOtherLanguageEntries`/`getActiveSubtypeString`/`switchToSubtypeString` + `langOf`/`layoutOf`; the `SwitcherShortcut` catalog + `layoutSwitcherShortcutPrefKey` + `getLayoutSwitcherShortcuts`; imports `PreferenceUtils`), `…/latin/LatinIME.kt` (the recency hook in the `ActiveSubtype` observer), `…/latin/settings/Settings.java` (the mode constant), `…/latin/uix/settings/pages/Typing.kt` (the spacebar-mode picker option + searchTag, **and** the "Layout switcher shortcuts" toggle section in `KxkbSizingScreen` — imports `SettingToggleSharedPrefs` + the catalog symbols), `java/res/values/strings-uix.xml` (the mode label).

When a tap's primary key shares its rect with the cell's other mains (cluster detected), the proximity list collapses to ONLY the co-located mains and `outIsExactSet[i] = 1`. `ProximityInfoState` carries `int mInputIsExactSet[MAX_WORD_LENGTH]`; `getProximityType` consults it at the top, returning MATCH for any listed main and **UNRELATED** (not SUBSTITUTION) for off-band keys — locks the dict's tap exploration in-set. After `decomposeTapPosition` builds the per-tap token mix (top-`NUM_TOKEN_MIX=4` letter tokens weighted by tap-circle overlap), find the dominant key (proportion > 0.5), get its `(keyX, keyY)` via NEW public `AK_FORCE_INLINE` `getKeyX/getKeyY` accessors on `ProximityInfo` (added in `proximity_info.h`), count co-located keys (same x,y) with positive proportion.   - **Kotlin merge — cluster-aware filter + bias + enumeration** (`LanguageModelFacilitator.kt`):
    - `clusterAllowedSets(values)`: returns `List<Set<Int>>?` of per-tap allowed lowercase codepoints. Per tap: `keyboard.getNearestKeys(x,y).firstOrNull{ it.isOnKey(x,y) }.clusterMains.map{ toLowerCase(it.codePoint) }.toHashSet()`. Returns `null` for gesture (`composedData.mIsBatchMode`), missing keyboard, or layouts with no cluster keys (filter doesn't apply). Non-cluster keys constrained to `{their own code}`; unresolved taps get an empty set (left unconstrained at that position).
    - `prefixInClusterSets(word, sets)`: prefix-not-exact-length so `simp → simple/simply` works (`word.length >= sets.size`, every position with a non-empty set must match by `Character.toLowerCase`). Applied to BOTH the LM suggestion list (filter call before the bias map) and the dict suggestion list (`mapNotNull` filter on `suggestedWordsDict.mSuggestedWordInfoList`, exempting `suggestedWordsDict.typedWordInfo`).
    - `clusterLengthBias(len, tapCount)`: `2.0f` for exact-length, `Math.pow(baseFactor, len-tapCount)` for completions, where **`baseFactor = if (tapCount == 4) 0.7 else 0.55`** — softer specifically for 4-tap prefixes so `simp → simple/simply` actually surfaces in the alternative slots (shorter inputs get exact-length matches injected by enumeration and don't need completion help; longer inputs are usually full-word typing where completions are noise rather than intent).
    - `clusterEnumerateValid(clusterSets)`: cartesian product of per-tap letter mains (symbols filtered via `Character.isLetter`), **capped at 2000 combinations** so per-keystroke cost stays bounded (typical 6-tap clusters are ~150 combinations). Each candidate checked via `dictionaryFacilitator.isValidSuggestionWord(word)`, returned as `List<Pair<String, Int>>` with `dictionaryFacilitator.getFrequency(word).coerceAtLeast(0)`.
    - Enumeration injection in `processAndMergeSuggestions` (after the bias `mapNotNull` builds `suggestedWordsDictList`): **dual freq threshold** — when the LM also picked the word it only needs `freq >= 30` (contextually confirmed by the LM); otherwise `>= 100` to cull proper-noun/abbreviation noise (`Diu`, `yip`, `doz`, …). On warmup with empty LM, a single `>= 50` floor. Survivors not already in the dict list are added as `KIND_CORRECTION` with score `(100M + freq*1M) * 2` — the 2× exact-length boost pre-applied since these are exact-length matches by construction (the bias `mapNotNull` runs only over the dict's original output, so injected entries skip it).
  - **Trade-offs settled with on-device testing.** Several rounds were tried over the user's battery (`the / you / yes / vet / cat / people / simple / simp`):
    - *Round A* — single `freq >= 50` filter, no LM signal. `vet` survives, but `cab > cat` in dict freq so the wrong word wins.
    - *Round B (shipped)* — dual `>= 30/>= 100` threshold. `cat > cab` works (cab fails the non-LM-confirmed gate), at the cost of `vet` disappearing in some contexts (not LM-confirmed there, freq below 100). User explicitly preferred this.
    - *Round C (tried, reverted)* — single `>= 50` floor + LM-confirmation BOOST on score (4× vs 2×) + proper-noun heuristic (skip if `getFrequency(capForm) > getFrequency(word)`). Brings `vet` back but the boost wasn't strong enough to flip `cab/cat` (cab also gets enumerated and freq-ranks ahead), and the merge re-ranking caused `simple` to lose its slot in the `simp` test.

(2) `Diu` (capitalised) can still appear from the dict's own native tap-decoder; the proper-noun-style filtering only ever applied to our enumeration's lowercase `diu` injection (and Round C's heuristic, since reverted). Both judged acceptable.
  - **Where to extend next** if this needs more work: rank-aware LM boost (top-N LM picks weighted more than tail), exposing `Dictionary.getMaxFrequencyOfExactMatches` separately from user-history freq, or per-position confidence weighting in the cartesian enumeration (tap centre vs edge).

- **Cluster prediction for accented / no-transformer languages (Czech, Russian) (LIVE)** (commit `bc1cd9e7c`). The cluster prediction above only worked for **English**. Two root causes, both found by on-device debugging (Huawei suppresses app logcat, so diagnostics were written to a file in `getExternalFilesDir` and pulled with `adb pull /sdcard/Android/data/shiroikuma.futokxkb/files/…` — the working pattern when logcat is dead). All fixes are **language-agnostic, Kotlin-only (2 files), no native change**:
  - **Root cause 1 — the whole pipeline was transformer-gated.** The cluster filter + enumeration live in `LanguageModelFacilitator.processAndMergeSuggestions`, which `GeneralIME` only calls when the transformer LM returns results (`!lmResult.isNullOrEmpty()`). Czech/Russian have a **dictionary but no transformer model** (`ModelPaths.getModelOptions` has no entry → `runLanguageModel` sets `skipLanguage` → `shouldPassThroughToLegacy()` true → `makePredictionInputValues` returns null → the dict-only legacy branch runs, with **none** of the cluster code). English worked only because it *has* a transformer model. Fix: **`maybeApplyClusterConstraint(inputStyle, dictResult)`** — called from `GeneralIME`'s `dictResult != null` (legacy) branch — builds values via `makePredictionInputValues(…, ignorePassthrough = true)` (new param that skips the passthrough guard), and if the layout is a cluster layout routes the raw dict result back through `processAndMergeSuggestions` with an **empty** LM list. Returns null (caller keeps the untouched dict result) for non-cluster layouts, so ordinary legacy typing is unaffected.
  - **Root cause 2 — accent-blindness.** Cluster mains are base-Latin/Cyrillic but the dictionary stores accented words, and `isValidSuggestionWord`/`getFrequency` are **accent-exact**; the dict decoder only ever surfaced wrong neighbours (`těžce`, never `těžko`). Fix: **`clusterBaseFold(cp)`** — `Character.toLowerCase` then (non-ASCII only) NFD-decompose + drop combining marks → base letter (`ě/é→e`, `ž→z`, `ů→u`; Cyrillic `ё→е`, `й→и`), mirroring the native `toBaseLowerCase`/`BASE_CHARS`. Used in `clusterAllowedSets` (folds the mains) and `prefixInClusterSets` (folds the candidate char). ASCII fast-path → **English byte-for-byte unchanged**.
  - **The enumeration is now a dictionary trie-walk, not a cartesian product** — `clusterTrieWalk(clusterSets, accentVariants)`. KEY DISCOVERY: `Dictionary.getValidNextCodePoints(ComposedData)` already exists (aggregated across dicts by `DictionaryFacilitator.getValidNextCodePoints`, which was **already on the interface** — no new method needed) and returns the valid next code points after a prefix. The walk: at each tap, get the dict's valid continuations of the prefix built so far and follow only those whose `clusterBaseFold` is in that tap's cluster set. This **prunes to real dictionary paths** (the 10-tap `jednoduché` returns only the 2 valid in-set words — no blow-up), yields the **accented spelling straight from the trie** (no per-language accent tables), and has no length cap. **GOTCHA:** `getValidNextCodePoints("")` returns nothing (the query needs a non-empty prefix), so **position 0 is seeded** from the cluster's own letters + their accent variants (harvested from the dict's near-candidates) rather than a dict query; positions ≥1 use the dict. The old cartesian+accent-harvest enumeration (`clusterEnumerateValid` + `accentVariantsFromDict`) is **kept as a fallback** when the walk returns nothing.
  - **Confident-cluster auto-commit.** On a cluster layout the "typed word" is the meaningless centre-letter commit (`teuwi` for `těžko`), so AOSP's normalized-score-vs-typed-word autocorrect test in `Suggest.obtainNonBatchedInputSuggestedWords` always fails (huge edit distance) → the predicted word sat in slot 2 and space committed the literal. Fix: for cluster layouts with a confident in-set word AND **≥ 4 taps** (shorter inputs stay suggest-only to avoid mis-commit), pass `firstSuggestionExceedsConfidenceThreshold = true` to the `SuggestionResults` constructor (computed up front since that flag is constructor-set/final), so the top in-set word becomes the auto-commit. **Tuning knobs if needed:** the `>= 4` tap threshold (commit aggressiveness) and the `>= 30/100/50` enumeration freq floors.
  - **Why the earlier "fix" only half-worked:** `těžko`/`daleko` (short/medium) succeeded once the fold + legacy-path + accent enumeration landed; `jednoduché` (10 taps) failed until the trie-walk replaced the cartesian (which exceeded its size cap). A regression round where the trie-walk returned empty for *every* word traced to the seed-position-0 gotcha above.
  - Verified on-device for Czech (`těžko`, `daleko`, `jednoduché`) and Russian. 2 files: `LanguageModelFacilitator.kt` (`clusterBaseFold`, `clusterTrieWalk` + `clusterWalkPointers`, `accentVariantsFromDict`/`clusterEnumerateValid` fallback, `maybeApplyClusterConstraint`, `makePredictionInputValues` `ignorePassthrough`, the hoisted enumeration + `clusterConfidentAutocorrect`), `GeneralIME.kt` (the legacy-branch wiring).
